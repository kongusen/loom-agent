# 能力边界

一个健康的 wiki 不能只写“系统能做什么”，还要写清楚“系统不能假装已经做到什么”。

## 边界一：设计目标不等于现有能力

| 主题 | 设计定义 | 当前状态 |
|---|---|---|
| 独立心跳感知层 `H_b` | 与主循环并行运行，解决等待盲区 | `heartbeat` 模块存在，但整体实现仍偏早期 |
| 多 Agent 自主协作 | 任务拆分、共享上下文、结构化回传 | 有 `SubAgentManager`、事件总线等基础能力，协作深度仍可继续加强 |
| Skill / Plugin / MCP 生态 | 渐进式加载、动态替换、外部能力注入 | 已有生态目录和加载结构，但生态成熟度仍在建设 |
| 运行时 API | 面向开发者的统一入口 | `loom/api/` 已存在，表面较清晰，内部仍有演化空间 |

## 边界二：当前代码里存在“架构方向已确定，但实现仍在收敛”的迹象

从源码可以看到一些重复定义和过渡痕迹，例如：

- `ContextManager` 中有重复方法定义。
- `Loop` 中也出现过渡阶段的重复 `step` 定义。
- 部分模块同时存在“设计态命名”和“实现态命名”。

这说明项目正处在快速重构阶段。wiki 应该忠实反映这种状态，而不是把所有设计目标写成“已经稳定可用”。

## 边界三：对外叙述要克制

对外表达时建议使用以下分层措辞：

- 已经稳定存在的能力：直接陈述。
- 已经有代码承接但还在演进的能力：写成“已具备基础实现”。
- 主要来自 hernss 设计但尚未完全落地的能力：写成“设计目标”或“架构方向”。

## 建议的文档标注规范

后续新增页面时，可以统一采用这三类标签：

- `已实现`
- `部分实现`
- `设计目标`

这样就能同时服务产品表达、架构设计和代码事实。
