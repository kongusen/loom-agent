# 产品定位

Loom 不只是“一个能调模型的 SDK”，它更准确的定位是：

> 面向长周期、复杂任务和多能力协作场景的 Agent Runtime Framework。

## 为什么需要 Loom

大多数 Agent 系统在任务变长后会暴露三个问题：

- 上下文快速膨胀，认知质量下降。
- 工具调用增多后，可控性和可观测性变差。
- 一旦需要分工、重试、恢复或外部感知，系统会退化成脆弱脚本。

Loom 想解决的不是“让模型再聪明一点”，而是“给 Agent 一个更稳的运行时结构”。

## 价值主张

| 价值 | 对外表达 | 代码事实支撑 |
|---|---|---|
| 长任务稳定性 | 问题变长时不容易崩 | `loom/context/`、`loom/execution/`、`loom/memory/` |
| 多能力协作 | 一个系统内协调工具、子 Agent、生态能力 | `loom/tools/`、`loom/orchestration/`、`loom/ecosystem/` |
| 安全可控 | 权限、约束、审计有明确位置 | `loom/safety/` |
| 可持续演化 | 不把所有能力焊死在单一 Agent 上 | `loom/capabilities/`、`loom/evolution/` |

## 与普通 Agent 封装的区别

| 类型 | 常见特点 | Loom 的差异 |
|---|---|---|
| Prompt 封装器 | 快速、轻量，但难以扩展 | Loom 更重视运行时结构 |
| 工具编排器 | 流程清晰，但对非线性任务脆弱 | Loom 允许任务分裂、恢复、演化 |
| Workflow Engine | 强控制，弱自主 | Loom 尝试平衡自主性与边界约束 |

## 对外叙述建议

建议统一使用以下表述：

- “Loom 是面向复杂任务的 Agent Runtime Framework”
- “它强调上下文控制、运行时控制、工具治理、多 Agent 协作和安全边界”
- “它已经具备清晰的代码骨架，部分高级能力仍在持续收敛”
