# 扩展开发

Loom 的扩展点主要集中在运行时 API、 Provider、 工具和生态四个方向。## 方向一：扩展运行时 API

优先查看：
- `loom/api/runtime.py`
- `loom/api/handles.py`
- `loom/api/models.py`
- `loom/api/events.py`
- `loom/api/artifacts.py`
适合做的扩展：
- 新的会话、任务、运行控制接口
- 更细的事件模型
- 更完整的审批与外部等待语义
- 更稳定的 transcript / artifact 能力
## 方向二:扩展 Provider
优先查看：
- `loom/providers/base.py`
- `loom/providers/openai.py`
- `loom/providers/anthropic.py`
- `loom/providers/gemini.py`
适合做的扩展：
- 新模型提供方接入
- 更完整的 `complete / stream` 语义
- token usage 和错误模型补全
## 方向三:扩展工具
优先查看：
- `loom/tools/base.py`
- `loom/tools/schema.py`
- `loom/tools/registry.py`
- `loom/tools/builtin/`
适合做的扩展：
- 新增内置工具
- 增加工具治理规则
- 补充资源分配策略
## 方向四:扩展生态
优先查看：
- `loom/api/profile.py`
- `loom/api/knowledge.py`
- `loom/ecosystem/`
- `loom/capabilities/`
适合做的扩展：
- Skill 加载器
- Plugin 集成层
- MCP 资源桥接

## 扩展时要注意什么
| 原则 | 原因 |
|---|---|
| | 优先扩展 `loom/api` 的对象语义 | 否则外部接入层会持续漂移 |
| | 不要把扩展逻辑直接塞进核心闭环 | 否则会破坏边界清晰度 |
| | 先确认能力属于哪一层 | 否则目录会重新混乱 |
| | 文档里同步标注实现状态 | 避免设计与代码再次脱节 |
