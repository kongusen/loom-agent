# 开发者工作流
Loom 不是那种"读完一个 README 就能完全吃透"的项目。当前更高效的方式是先按 API 对象，再按模块维度进入。

## 娡块索引

| 需求类型 | 优先查看 |
|---|---|
| 会话、任务、运行控制 | `loom/api/` |
| Agent 内核、约束、主循环 | `loom/agent/`、`loom/execution/` |
| 上下文管理 | `loom/context/` |
| 工具能力 | `loom/tools/` |
| 子 Agent / 协作 | `loom/orchestration/`、`loom/cluster/` |
| 权限与约束 | `loom/safety/` |
| 生态接入 | `loom/ecosystem/` |

## 寏次开发先判断三件事

```mermaid
flowchart TD
    Start["开始开发"] --> Judge1{"1. 这是 API 层还是核心层？"}
    Judge1 -->|"API"| Judge2["2. 这是设计目标还是实现？"]
    Judge1 -->|"核心"| Judge2

    Judge2 -->|"设计目标"| Plan["补足设计目标"]
    Judge2 -->|"实现"| Judge3["3. 文档是否需要同步更新？"]
    Judge3 -->|"是"| UpdateWiki["更新 wiki"]
    Judge3 -->|"否"| Code["继续开发"]

    Plan --> Code
    Code --> UpdateWiki

    UpdateWiki --> End["完成"]

    style Judge1 fill:#D4850A,color:#fff
    style Judge2 fill:#2D8659,color:#fff
    style Judge3 fill:#2D8659,color:#fff
```

1. **先判断你改的是 API 层还是核心层**
2. **再判断"目标是补实现，还是修正表达"**
3. **最后确认文档是否需要同步更新**

## 测试建议

| 场景 | 优先查看 |
|---|---|
| 接口变更 | `tests/api/` |
| 工具/上下文/运行时变更 | `tests/core/` |
| 跨模块链路变更 | `tests/integration/` |

## 文档同步建议

只要出现下面三种情况，就应该同步更新 wiki：

- **模块边界变了** — 新增或重命名了目录
- **能力状态变化** — 从 `部分实现` 単级到了 `已实现`
- **对外说法不再准确** — 文档描述与代码事实不符
